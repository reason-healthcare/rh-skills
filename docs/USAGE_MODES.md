# Usage Modes

The RH Skills supports two usage modes. Both share the same guiding principle:

> **All deterministic work via `rh-skills` CLI. All reasoning by the agent.**

The `rh-skills` CLI handles file I/O, checksums, schema validation, tracking, and artifact promotion. The agent — whether that's a human at a terminal or an AI assistant — handles reasoning: what to extract, how to interpret a guideline, which sources are relevant.

---

## Mode 1: CLI-First

The user drives the workflow by calling `rh-skills` commands directly. An LLM is invoked only for the reasoning steps (plan → implement → verify), and the user controls when and how that happens.

```
User → rh-skills commands → artifacts + tracking.yaml
                  ↑
          LLM called by user
          (own choice of model/provider)
```

**When to use this mode:**
- You want full control over every step
- You want to use your own LLM or a locally-hosted model (Ollama, LM Studio, etc.)
- You're scripting or automating workflows in CI/CD
- You prefer reproducible, auditable pipelines
- You're developing or testing skills

**Key benefits:**
- **Bring your own model** — configure any LLM provider in `.env`; the `rh-skills` CLI is model-agnostic
- **No agent platform required** — works with any tool that can run shell commands
- **Full transparency** — every action is explicit; nothing happens without a deliberate command
- **Offline-capable** — deterministic steps (`rh-skills init`, `rh-skills list`, `rh-skills validate`, `rh-skills status`) require no LLM at all

**Quickstart:**

```bash
# Configure your preferred LLM provider
# .rh-skills.toml (local) or ~/.rh-skills.toml (global)
# [llm]
# provider = "ollama"
# Or use environment variables: LLM_PROVIDER, OLLAMA_MODEL, etc.

# Initialize a topic
rh-skills init diabetes-screening --title "Diabetes Screening"

# Run the plan step (calls your LLM via rh-skills)
rh-skills ingest plan diabetes-screening

# Review and approve the plan, then implement
rh-skills ingest implement diabetes-screening --source ada-2024-guidelines.pdf

# Check status at any point
rh-skills status show diabetes-screening
rh-skills status next-steps diabetes-screening
```

See [docs/WORKFLOW.md](WORKFLOW.md) for the full step-by-step lifecycle.

---

## Mode 2: Agent-Native

The RH skills are installed into the user's AI agent environment (GitHub Copilot, Claude Code, Gemini CLI, etc.). The user interacts naturally with their agent; the agent reads the skill instructions and calls `rh-skills` commands on the user's behalf.

```
User (natural language) → Agent (reads SKILL.md) → rh-skills commands → artifacts + tracking.yaml
```

**When to use this mode:**
- You already use an AI agent (Copilot, Claude, Gemini) as your primary interface
- Your clinical team is less familiar with CLI workflows
- You want conversational guidance through the L1→L2→L3 lifecycle
- You want the agent to handle sequencing decisions ("what should I do next?")

**Key benefits:**
- **Natural language UX** — "extract clinical criteria from the ADA guideline" rather than typing command flags
- **Agent handles sequencing** — the skill instructs the agent which `rh-skills` commands to run and when
- **Same `rh-skills` CLI underneath** — identical outputs; no special behavior or separate code path
- **Works with your existing agent** — no new tooling; install skills into the platform you already use

**Supported platforms:**

| Platform | Status | Install location |
|----------|--------|-----------------|
| GitHub Copilot | ✅ Supported | `.agents/skills/` |
| Claude Code (`claude` CLI) | ✅ Supported | `.claude/commands/` |
| Gemini CLI | 🔜 Planned | `.gemini/` |

Skills for each platform are generated by the build system — see [DEVELOPER.md](../DEVELOPER.md) for how to build and install them.

**Quickstart (GitHub Copilot):**

```bash
# Build and install skills for Copilot
scripts/build-skills.sh --platform copilot

# Skills are now available in .agents/skills/
# Use your agent as normal — it will pick up the RH skills automatically
```

## Comparing the Modes

| | CLI-First | Agent-Native |
|-|-----------|-------------|
| **Primary interface** | Terminal (`rh-skills` commands) | AI agent (natural language) |
| **LLM choice** | Any provider you configure | Determined by your agent platform |
| **Control** | Explicit — you run each step | Delegated — agent sequences steps |
| **Reproducibility** | High — same command = same output | Model-dependent |
| **Transparency** | Full — every command is visible | Agent reasoning is implicit |
| **Offline use** | Partial (deterministic steps only) | Requires agent connection |
| **CI/CD suitability** | ✅ Yes | ⚠️ Limited |
| **Barrier to entry** | Moderate (CLI familiarity) | Low (conversational) |
| **`rh-skills` CLI required** | ✅ Yes | ✅ Yes (called by agent) |

---

## Using Both Together

The modes are complementary, not exclusive. A common pattern:

1. **Use CLI-first** during development and testing — full control, easy debugging
2. **Use Agent-native** when working with clinical subject matter experts — lower barrier, more conversational
3. **Use CLI-first** in CI/CD for automated validation (`rh-skills validate`, `rh-skills status check-changes`)

Because both modes produce identical outputs via the same `rh-skills` CLI, artifacts and `tracking.yaml` are fully interchangeable between modes.

---

## LLM Provider Configuration (CLI-First)

In CLI-first mode, the `rh-skills` CLI invokes your configured LLM provider for
reasoning steps. Configure via `.rh-skills.toml` or environment variables:

```toml
# .rh-skills.toml (local) or ~/.rh-skills.toml (global)
[llm]
provider = "ollama"            # ollama | anthropic | openai

# Ollama (local, no API key required)
# endpoint = "http://localhost:11434"
# model = "mistral"

# Anthropic
# api_key = "sk-ant-..."
# model = "claude-3-5-sonnet-20241022"

# OpenAI-compatible
# api_key = "sk-..."
# endpoint = "https://api.openai.com/v1/chat/completions"
# model = "gpt-4o-mini"
```

```bash
# Or via environment variables (take precedence over .rh-skills.toml)
export LLM_PROVIDER=ollama
export OLLAMA_ENDPOINT=http://localhost:11434
export OLLAMA_MODEL=mistral
```

Any OpenAI-compatible endpoint works — local models via Ollama, LM Studio, vLLM, or hosted providers. The `rh-skills` CLI is model-agnostic; all reasoning is in the SKILL.md prompts, not hardcoded to any model.
