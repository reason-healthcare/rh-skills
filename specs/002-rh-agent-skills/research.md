## Decision 1: SKILL.md Structure for Multi-Modal Skills

**Decision**: Use `$ARGUMENTS` positional convention with **narrative conditional sections** in the SKILL.md body. No frontmatter `mode` field — branching is determined by reading `$ARGUMENTS` at runtime and executing the corresponding section.

**Rationale**: Existing speckit skills use exactly this pattern — conditional behavior is expressed as imperative Markdown narrative ("If $ARGUMENTS is `plan`, do X; if `implement`, do Y"). The agent interprets and routes. This keeps SKILL.md files simple and consistent with the existing ecosystem.

**Pattern** (per existing skills in `.agents/skills/`):
```markdown
---
name: "rh-inf-extract"
description: "Extract structured artifacts from sources. Modes: plan | implement | verify"
compatibility: "rh-skills >= 0.1.0"
metadata:
  author: "Clinical Informatics Team"
  source: "skills/.curated/rh-inf-extract/SKILL.md"
---

## User Input

```text
$ARGUMENTS
```

You **MUST** consider the mode in $ARGUMENTS before proceeding.
Valid modes: `plan` | `implement` | `verify`
If no mode is provided, print usage and stop.

## Mode: plan
[instructions for plan mode...]

## Mode: implement
[instructions for implement mode...]

## Mode: verify
[instructions for verify mode...]
```

**Required frontmatter fields**: `name`, `description`, `compatibility`, `metadata.author`, `metadata.source`

**Alternatives considered**:
- Separate SKILL.md per mode (~14 files) — rejected: too many files, harder to maintain consistency
- Frontmatter `modes[]` array — rejected: no existing skills use this; agents ignore frontmatter beyond name/description

## Decision 2: Plan Artifact Format (YAML Front Matter + Markdown)

**Decision**: Plan artifacts are Markdown files with a YAML front matter block between `---` delimiters. The front matter contains machine-readable fields; the prose body is for human review.

**Rationale**: `implement` mode must reliably extract artifact names, source files, and configuration from the plan without LLM reasoning. YAML front matter provides a deterministic, structured extraction target while keeping the file human-readable and editable.

**Format**:
```markdown
---
topic: diabetes-screening
plan_type: extract
version: "1.0"
created: "2026-04-03T19:00:00Z"
artifacts:
  - name: screening-criteria
    source_file: ada-guidelines-2024
    description: "Discrete criteria for who should be screened"
  - name: risk-factors
    source_file: ada-guidelines-2024
    description: "Enumerated risk factor definitions"
---

## Extract Plan — diabetes-screening

This plan was generated by `rh-inf-extract plan` on 2026-04-03.
**Review and edit the YAML front matter before running `rh-inf-extract implement`.**

### Proposed Structured Artifacts

#### screening-criteria
Derived from: `ada-guidelines-2024`
...human prose describing what this artifact will contain...
```

**Python extraction of front matter** (using `ruamel.yaml`):
```python
import re
from ruamel.yaml import YAML
from pathlib import Path

def parse_plan_artifact(path: Path) -> dict:
    """Extract YAML front matter from a plan artifact."""
    content = path.read_text()
    m = re.match(r'^---\n(.*?\n)---\n', content, re.DOTALL)
    if not m:
        raise ValueError(f"No YAML front matter found in {path}")
    yaml = YAML()
    return dict(yaml.load(m.group(1)))
```

This utility lives in `src/hi/common.py` and is imported by command modules.

**Alternatives considered**:
- Pure YAML plan file — rejected: not human-readable/editable without knowing YAML syntax
- Separate YAML + Markdown files — rejected: two files to keep in sync; user might edit one but not the other
- LLM re-reads prose plan — rejected: non-deterministic; violates "deterministic work to commands" principle

## Decision 3: Skill File Location

**Decision**: `skills/.curated/<skill-name>/SKILL.md` — inside the skills tree, dot-prefixed directory to signal "curated framework" rather than a clinical skill.

**Rationale**: Keeps framework skills visible alongside clinical skills without polluting the clinical namespace. The `.curated/` directory uses the dot-prefix convention and is excluded from topic listings.

**Directory layout**:
```
skills/
  .curated/             ← framework agent skills (not clinical skills)
    rh-inf-discovery/
    rh-inf-ingest/
    rh-inf-extract/
    rh-inf-formalize/
    rh-inf-verify/
    rh-inf-status/
topics/
  diabetes-screening/   ← clinical topics (listed by rh-skills list)
  sepsis-detection/
```

**Alternatives considered**:
- `.agents/skills/` — rejected: that directory belongs to speckit framework, not RH skills
- `skills/_framework/` — rejected: underscore prefix is not standard for this project
- Per-clinical-skill copies — rejected: maintenance nightmare; skills would diverge

## Decision 4: SHA-256 Checksum

**Decision**: SHA-256 computation uses Python's `hashlib` module — no external tools required.

**Implementation**:
```python
import hashlib
from pathlib import Path

def sha256_file(path: Path) -> str:
    """Compute SHA-256 hash of a file."""
    h = hashlib.sha256()
    with open(path, "rb") as f:
        for chunk in iter(lambda: f.read(65536), b""):
            h.update(chunk)
    return h.hexdigest()
```

**Checksum storage in tracking.yaml** (appended to ingest records):
```yaml
sources:
  - name: ada-guidelines-2024
    file: sources/ada-guidelines-2024.md
    type: document
    ingested_at: "2026-04-03T19:00:00Z"
    checksum: "e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855"
    text_extracted: false
```

**Alternatives considered**:
- Store checksums in a separate `.checksums` file — rejected: tracking.yaml is the single source of truth
- MD5 — rejected: SHA-256 is standard for integrity, MD5 has collisions
- Shell `sha256sum`/`shasum` — rejected: replaced by Python hashlib (no external tool dependency)

## Decision 5: Optional Tool Handling (pdftotext, pandoc)

**Decision**: Detect at invocation time with `shutil.which`. If absent: register file metadata + checksum but skip text extraction, emit a warning to stderr, exit 0 (not failure). Record `text_extracted: false` in tracking.yaml.

**Python implementation**:
```python
import shutil

def check_optional_tool(tool: str, package: str, context: str) -> bool:
    """Check if an optional tool is available, warn if not."""
    if shutil.which(tool) is None:
        log_warn(f"{tool} not found — {context} will be skipped")
        log_warn(f"Install: brew install {package} (macOS) or apt install {package} (Linux)")
        return False
    return True
```

**Alternatives considered**:
- Fail hard if pdftotext missing — rejected: breaks ingest for teams that only use Markdown/text sources
- Always require pandoc and pdftotext — rejected: no new mandatory dependencies

## Decision 6: Re-run / Idempotency with --force

**Decision**: If a plan or implement output already exists, print a clear warning with the existing file path and suggest `--force`. With `--force`, overwrite silently. This applies to both plan files (`.md`) and implemented artifacts (structured/computable YAML).

**Python implementation** (using click):
```python
@click.option("--force", is_flag=True, help="Overwrite existing output")
def plan(force: bool):
    if plan_file.exists() and not force:
        log_warn(f"Plan already exists: {plan_file}")
        log_warn("Review it or re-run with --force to overwrite")
        return
```

**Alternatives considered**:
- Always overwrite — rejected: risk of losing reviewed/edited plans
- Append/merge — rejected: complex to implement reliably; edited plans may conflict with merged content

## Decision 7: New CLI Module (rh-skills ingest)

**Decision**: Add `src/hi/commands/ingest.py` as a new click command group. It provides `plan`, `implement`, and `verify` subcommands. Checksum and file registration helpers are in `src/hi/common.py`.

**Rationale**: The ingest skill needs deterministic file operations (checksum, YAML write, file type detection) that must live in CLI code, not LLM prompts. The existing click command group pattern applies unchanged.

**Alternatives considered**:
- Implement ingest entirely within SKILL.md prose — rejected: file I/O and checksum are deterministic, must be in CLI
- Extend `rh-skills promote` to handle ingest — rejected: `promote` is for L2→L3 progression; ingest is source registration
