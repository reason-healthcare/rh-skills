## Decision 1: SKILL.md Structure for Multi-Modal Skills

**Decision**: Use `$ARGUMENTS` positional convention with **narrative conditional sections** in the SKILL.md body. No frontmatter `mode` field — branching is determined by reading `$ARGUMENTS` at runtime and executing the corresponding section.

**Rationale**: Existing speckit skills use exactly this pattern — conditional behavior is expressed as imperative Markdown narrative ("If $ARGUMENTS is `plan`, do X; if `implement`, do Y"). The agent interprets and routes. This keeps SKILL.md files simple and consistent with the existing ecosystem.

**Pattern** (per existing skills in `.agents/skills/`):
```markdown
---
name: "hi-extract"
description: "Extract L2 artifacts from L1 sources. Modes: plan | implement | verify"
compatibility: "hi-skills-framework >= 0.1.0"
metadata:
  author: "Clinical Informatics Team"
  source: "skills/_framework/hi-extract/SKILL.md"
---

## User Input

```text
$ARGUMENTS
```

You **MUST** consider the mode in $ARGUMENTS before proceeding.
Valid modes: `plan` | `implement` | `verify`
If no mode is provided, print usage and stop.

## Mode: plan
[instructions for plan mode...]

## Mode: implement
[instructions for implement mode...]

## Mode: verify
[instructions for verify mode...]
```

**Required frontmatter fields**: `name`, `description`, `compatibility`, `metadata.author`, `metadata.source`

**Alternatives considered**:
- Separate SKILL.md per mode (~14 files) — rejected: too many files, harder to maintain consistency
- Frontmatter `modes[]` array — rejected: no existing skills use this; agents ignore frontmatter beyond name/description

## Decision 2: Plan Artifact Format (YAML Front Matter + Markdown)

**Decision**: Plan artifacts are Markdown files with a YAML front matter block between `---` delimiters. The front matter contains machine-readable fields; the prose body is for human review.

**Rationale**: `implement` mode must reliably extract artifact names, source files, and configuration from the plan without LLM reasoning. YAML front matter provides a deterministic, structured extraction target while keeping the file human-readable and editable.

**Format**:
```markdown
---
skill: diabetes-screening
plan_type: extract
version: "1.0"
created: "2026-04-03T19:00:00Z"
artifacts:
  - name: screening-criteria
    source: ada-guidelines-2024
    description: "Discrete criteria for who should be screened"
  - name: risk-factors
    source: ada-guidelines-2024
    description: "Enumerated risk factor definitions"
---

## Extract Plan — diabetes-screening

This plan was generated by `hi-extract plan` on 2026-04-03.
**Review and edit the YAML front matter before running `hi-extract implement`.**

### Proposed L2 Artifacts

#### screening-criteria
Derived from: `ada-guidelines-2024`
...human prose describing what this artifact will contain...
```

**Bash extraction of front matter** (bash 3.2+ portable — confirmed on bash 3.2.57):
```bash
# Extract YAML front matter block (between first and second ---)
# Two methods — both portable; sed preferred for single-pass simplicity
hi_markdown_get_frontmatter_block() {
  local md_file="$1"
  [[ ! -f "$md_file" ]] && return 1
  local c
  c=$(grep -c "^---$" "$md_file" || echo 0)
  [[ $c -lt 2 ]] && return 1
  sed -n '1,/^---$/p' "$md_file" | sed '1d;$d'
}

# Read a single front matter field
hi_markdown_get_field() {
  local md_file="$1" field="$2" front val
  front=$(hi_markdown_get_frontmatter_block "$md_file") || return 1
  val=$(echo "$front" | yq eval ".$field" - 2>/dev/null)
  [[ "$val" == "null" || -z "$val" ]] && echo "" || echo "$val"
}

# Get prose body (everything after the closing ---)
hi_markdown_get_content() {
  local md_file="$1" line
  [[ ! -f "$md_file" ]] && return 1
  line=$(grep -n "^---$" "$md_file" | tail -1 | cut -d: -f1)
  [[ -z "$line" ]] && return 1
  tail -n +$((line + 1)) "$md_file"
}

# Usage examples:
# front=$(hi_markdown_get_frontmatter_block "$plan_file")
# artifacts=$(echo "$front" | yq eval '.artifacts[].name' -)
# plan_type=$(hi_markdown_get_field "$plan_file" 'plan_type')
```

**These three functions go in `bin/common.sh`** and are sourced by `bin/hi-ingest` and any skill that needs to read plan artifacts.

**yq syntax confirmed**: Project uses yq v4.x (`yq eval` — not legacy `yq e` or v3 `yq read/write`). Stdin piping: `echo "$yaml" | yq eval '.field' -`.

**Alternatives considered**:
- Pure YAML plan file — rejected: not human-readable/editable without knowing YAML syntax
- Separate YAML + Markdown files — rejected: two files to keep in sync; user might edit one but not the other
- LLM re-reads prose plan — rejected: non-deterministic; violates "deterministic work to commands" principle

## Decision 3: Skill File Location

**Decision**: `skills/_framework/<skill-name>/SKILL.md` — inside the skills tree, underscore-prefixed to signal "internal/framework" rather than a clinical skill.

**Rationale**: Keeps framework skills visible alongside clinical skills without polluting the clinical namespace. The `_framework/` prefix follows Unix convention for "private" or "system" directories and is trivially excluded from `hi list` by checking for the `_` prefix.

**Directory layout**:
```
skills/
  _framework/           ← framework agent skills (not clinical skills)
    hi-discovery/
    hi-ingest/
    hi-extract/
    hi-formalize/
    hi-verify/
    hi-status/
  diabetes-screening/   ← clinical skills (listed by hi list)
  sepsis-detection/
```

**Alternatives considered**:
- `.agents/skills/` — rejected: that directory belongs to speckit framework, not HI skills
- `bin/skills/` — rejected: `bin/` is for executable scripts, not SKILL.md prompts
- Per-clinical-skill copies — rejected: maintenance nightmare; skills would diverge

## Decision 4: SHA-256 Checksum Portability

**Decision**: Runtime detection of `sha256sum` (Linux) vs `shasum -a 256` (macOS/BSD) via a portable wrapper function. Use simple command detection (not associative arrays, which require bash 4+).

**Portable implementation** (bash 3.2 compatible):
```bash
# Compute SHA-256 hash — works on macOS and Linux
hi_sha256() {
  local file="$1"
  if command -v sha256sum >/dev/null 2>&1; then
    sha256sum "$file" | awk '{print $1}'
  elif command -v shasum >/dev/null 2>&1; then
    shasum -a 256 "$file" | awk '{print $1}'
  else
    hi_log_error "No SHA-256 tool found (need sha256sum or shasum)"
    return 1
  fi
}
```

**Checksum storage in tracking.yaml** (appended to ingest records):
```yaml
sources:
  - name: ada-guidelines-2024
    path: skills/diabetes-screening/l1/ada-guidelines-2024.md
    type: markdown
    ingested_at: "2026-04-03T19:00:00Z"
    checksum: "e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855"
    text_extracted: true
```

**Alternatives considered**:
- Store checksums in a separate `.checksums` file — rejected: tracking.yaml is the single source of truth; splitting state creates consistency risk
- MD5 — rejected: SHA-256 is standard for integrity, MD5 has collisions

## Decision 5: Optional Tool Handling (pdftotext, pandoc)

**Decision**: Detect at invocation time with `command -v`. If absent: register file metadata + checksum but skip text extraction, emit a `[WARN]` to stderr, exit 0 (not failure). Record `text_extracted: false` in tracking.yaml.

**Bash 3.2 portable detection** (no associative arrays):
```bash
hi_require_optional_tool() {
  local tool="$1" package="$2" context="$3"
  if ! command -v "$tool" >/dev/null 2>&1; then
    hi_log_warn "$tool not found — $context will be skipped"
    hi_log_warn "Install: brew install $package (macOS) or apt install $package (Linux)"
    return 1
  fi
  return 0
}

# Usage:
if hi_require_optional_tool "pdftotext" "poppler" "PDF text extraction"; then
  pdftotext "$source_file" "$text_file"
fi
```

**Alternatives considered**:
- Fail hard if pdftotext missing — rejected: breaks ingest for teams that only use Markdown/text sources
- Always require pandoc and pdftotext — rejected: FR-024 requires no new mandatory dependencies

## Decision 6: Re-run / Idempotency with --force

**Decision**: If a plan or implement output already exists, print a clear warning with the existing file path and suggest `--force`. With `--force`, overwrite silently. This applies to both plan files (`.md`) and implemented artifacts (L2/L3 YAML).

**Implementation pattern**:
```bash
if [[ -f "$plan_file" ]] && [[ "${force:-0}" -eq 0 ]]; then
  hi_log_warn "Plan already exists: $plan_file"
  hi_log_warn "Review it or re-run with --force to overwrite"
  exit 0
fi
```

**Alternatives considered**:
- Always overwrite — rejected: risk of losing reviewed/edited plans
- Append/merge — rejected: complex to implement reliably; edited plans may conflict with merged content

## Decision 7: New CLI Command (hi-ingest)

**Decision**: Add `bin/hi-ingest` as a new CLI command following the existing dispatcher pattern. It provides `plan`, `implement`, and `verify` subcommands. Checksum and file registration helpers go in `bin/ingest-lib.sh` (sourced by `hi-ingest`).

**Rationale**: The ingest skill needs deterministic file operations (checksum, YAML write, file type detection) that must live in CLI scripts, not LLM prompts. The existing `bin/` pattern and exit code contract apply unchanged.

**Alternatives considered**:
- Implement ingest entirely within SKILL.md prose — rejected: file I/O and checksum are deterministic, must be in CLI
- Extend `hi promote` to handle ingest — rejected: `promote` is for L1→L2→L3 progression; ingest is source registration
